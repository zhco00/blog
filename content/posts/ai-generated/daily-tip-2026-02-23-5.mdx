# 코드 리뷰에서 감정적 피드백을 피하는 5가지 전략

## 개요
코드 리뷰는 개발자에게 가장 민감한 순간 중 하나입니다. 내가 며칠 밤을 새워 작성한 코드를 다른 사람이 평가하는 과정에서 감정적 충돌이 일어나기 쉽죠.

## 본문

### 1. "이 코드는 나쁘다" 대신 "이 접근법은 어떨까요?"

**❌ 피해야 할 표현:**
```
이 함수는 너무 복잡해요. 다시 작성하세요.
```

**✅ 개선된 표현:**
```
이 함수의 복잡도를 줄이기 위해 다음과 같이 분리해보는 건 어떨까요?

// Before
function processUserData(user) {
  // 50줄의 복잡한 로직
}

// After
function processUserData(user) {
  const validated = validateUser(user);
  const enriched = enrichUserData(validated);
  return formatUserOutput(enriched);
}
```

### 2. 구체적인 근거 제시하기

막연한 "좋지 않다"는 표현보다는 명확한 이유를 제시해야 합니다.

**Before:**
"이 방법은 별로예요."

**After:**
"현재 구현은 O(n²) 시간복잡도를 가지고 있어서, 데이터가 많아질 때 성능 이슈가 있을 수 있습니다. Map을 사용하면 O(n)으로 개선할 수 있어요."

### 3. 질문 형태로 피드백하기

명령형보다는 질문형 피드백이 더 수용하기 쉽습니다.

```
"이 부분에서 예외 처리를 추가하면 어떨까요? 
API 호출이 실패했을 때의 시나리오를 고려해보셨나요?"
```

### 4. 긍정적 요소 먼저 언급하기

비판적 피드백 전에 좋은 점을 먼저 언급하면 방어적 태도를 줄일 수 있습니다.

```
"함수 네이밍이 직관적이고 좋네요! 
다만 에러 핸들링 부분에서 사용자 경험을 더 개선할 수 있을 것 같아요."
```

### 5. 학습 기회로 전환하기

코드 리뷰를 지적의 시간이 아닌 함께 배우는 시간으로 만들어보세요.

```
"저도 최근에 비슷한 문제를 겪었는데, 
이 라이브러리를 사용해서 해결했어요. 
함께 검토해볼까요?"
```

### 실제 리뷰 예시

```python
# 개선 전 피드백
"이 코드 비효율적입니다."

# 개선 후 피드백
"좋은 로직이네요! 다만 반복문 안에서 데이터베이스 쿼리가 
여러 번 실행될 수 있어서, 배치 처리로 최적화하면 
성능이 크게 향상될 것 같아요.

# 현재 방식
for user in users:
    update_user_status(user.id)

# 개선 제안
user_ids = [user.id for user in users]
batch_update_user_status(user_ids)
```

### 팀 문화 만들기

개인적 노력뿐만 아니라 팀 차원에서도 건설적인 리뷰 문화를 만들어야 합니다. 정기적인 회고를 통해 리뷰 프로세스 자체를 개선하고, 서로의 커뮤니케이션 스타일을 이해하는 시간을 가져보세요.

## 마무리
- 비판보다는 제안과 질문 형태로 피드백을 전달하자
- 구체적인 근거와 대안을 함께 제시하여 건설적인 토론을 유도하자
- 코드 리뷰를 지적의 시간이 아닌 팀 전체가 성장하는 학습 기회로 만들자
